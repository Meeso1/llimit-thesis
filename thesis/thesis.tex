\section{API Interface}

In it's essence, the task completion process is very simple from user perspective. The only information that needs to be provided by the user is the task prompt, and files required for task completion. These actions were implemented as two \texttt{POST} endpoints. Supplementary \texttt{GET} endpoints on the same routes make it possible to inspect the state of running and completed tasks, completing the minimal API design required for task execution.

In practice, a few additional endpoints are required. The following routes were implemented:

\begin{itemize}
    \item \texttt{/tasks}: Creating new task, inspecting existing tasks;
    \item \texttt{/files}: Uploading a file with metadata, inspecting uploaded files;
    \item \texttt{/sse/events}: SSE event stream (described in section \ref{section:async-processing});
    \item \texttt{/allowed-models}: Setting a list of allowed LLMs for step completion (described in section \ref{section:model-restriction}).
\end{itemize}

Presented list does not describe all endpoints that were implemented. Routes not directly related to task completion, related to API key management etc., were omitted for brevity.

\section{Task Execution Process}

[TODO: diagram]

Diagram [TODO: ref] provides a high-level overview. Task completion process starts with user \textit{POST} request to \texttt{/tasks}, specifying the prompt, and a list of uploaded files that are required to successfully complete the task. Subsequently, step list is created based on the prompt. Steps are completed in order, based on dependencies between them. For each, a different LLM is selected to complete it, based on the prompt and estimated request cost (described in section \ref{section:llm-selection}). Some steps may define a planned re-evaluation point, in which case remaining step list is updated, based on the result of already completed steps. The same happens if step completion fails (section \ref{section:error-recovery}). Finally, the last step produces a text output, which is returned as the result of the task.

\subsection{Planning and Reevaluation}

TODO: some references etc.

Producing a set of steps required for completion of a given task can be very easily framed as a text processing task requiring a certain level of intelligence and reasoning. Therefore, the simplest solution to that problem is just using a sufficiently good, commercial LLM. While using a high-end LLM model may seem contradictory to the objective of reducing task completion cost, the decision is justified by the fact that step planning prompt is relatively short (compared to step prompts, which may contain much more information), and that the request doesn't contain any input images or files, which cause a dominant contribution to request cost. Planning shouldn't require too extensive reasoning, either, as the actual complex task completion work should be performed when steps are executed.

In initial planning request, the prompt contains task description specified by the user, along with a list of metadata of files attached to the task - their names, types, and user-specified descriptions. In case of reevaluation, outputs of already completed steps are also specified, along with failure reason of the task whose execution failed, if reevaluation was triggered by step failure. Requested output format is a JSON list of step definitions. Each step definition specifies a prompt, previous steps that it depends on (and whose outputs will be included when that step is processed), required files, and required LLM capabilities.

Requested output specifies a number of constraints that LLM response must satisfy, both defined by the format (JSON schema), and by semantic rules (step dependency graph must be acyclic, etc.). These constraints are validated when response is received. If the format is not correct, task status is set to failed and processing stops. While a retry mechanism could be easily implemented, it turned out to be unnecessary - modern commercial LLMs seem to be sufficiently capable of producing a structured output that follows such rules, making failures during planning step extremely rare.

\subsubsection{LLM Capabilities}

Different LLMs support different mechanisms and options - some allow image input, others implement native web search, etc. Those "hard" capabilities of models may be crucial for completion of certain steps, so they can be specified in step definition by planning model. Often, those capabilities serve as filters - if a step requires certain mechanisms or input modalities, LLMs that don't implement them cannot be used to complete that step. That's not always the case, though. \textit{OpenRouter} implements a range of simple options available regardless of requested model, which can serve as replacements for mechanisms that are not supported by a given LLM. These include PDF processing options, prompt-based web search, etc. If such capabilities are required by step definition, they don't cause any models to be filtered out, but instead cause a specific option to be included in the request, specifying usage of a certain mechanism.

The following capabilities are implemented:

\begin{itemize}
    \item \textbf{Reasoning}: requires an LLM that supports reasoning to be used, and requests medium-effort reasoning during step completion; 
    \item \textbf{EXA web search}: requests usage of prompt-based EXA web search during step completion;
    \item \textbf{Native web search}: requests usage of LLMs that support native web search (i.e. \textit{Gemini} and \textit{Perplexity} models); 
    \item \textbf{OCR PDF processing}: specifies that attached PDF files should be processed via OCR, using an \textit{OpenRouter}-managed mechanism;
    \item \textbf{Text-based PDF processing}: specifies that attached PDF files should be processed by simply extracting included text;
    \item \textbf{Native PDF processing}: requires usage of models that accept PDF as native model input;
    \item \textbf{Input modalities}: requires usage of models whose input modalities cover all types of attached files (implicitly specified by a list of files required for step completion).
\end{itemize}

\section{File Handling}

Files are uploaded independently from tasks. During upload, in addition to file contents and standard file upload metadata, request specifies declared content MIME type, optional description, and optional user-specified metadata.

The upload is followed by type-specific preprocessing. This step performs required type conversions, text extraction, and metadata computation. Text extraction is performed for supported Office XML formats (\texttt{.xlsx}, \texttt{.docx}) - as most LLMs don't support these formats as input, such files are converted to text using \texttt{docx} and \texttt{openpyxl} packages. 

As for metadata computation, the most important computed value is token count of text data. Depending on the format, text-based files are either priced as input tokens (i.e. PDFs if text-based processing is selected), or simply included in the prompt (i.e. plain-text or markdown files). This means that knowing the token count of their contents is required for accurate request cost prediction (described in detail in section \ref{section:cost-estimation}).

Computed metadata are appended to user-specified additional metadata. Finally, the file is saved to disk, (encoded as Base64 for security), and metadata is stored in the database.

In addition to actual files, system supports registering file URLs. LLMs generally allow to specify URLs as input in case of images, video, audio, and PDF files, so supporting this format in the system was a natural consequence. URL registration process is analogous to file upload, with exception of metadata computation. In this case, as file contents aren't available, data required for cost estimation is expected to be provided by the user in additional metadata.

The following file types are supported:

\begin{itemize}
    \item Image: \texttt{PNG}, \texttt{JPEG}, \texttt{GIF}, \texttt{WebP};
    \item Video: \texttt{MP4}, \texttt{MOV}, \texttt{MPEG}, \texttt{WebM};
    \item Audio: \texttt{MP3}, \texttt{MPEG}, \texttt{WAV};
    \item PDF files;
    \item Office formats: \texttt{XLSX}, \texttt{DOCX};
    \item JSON files;
    \item All text-based formats: MIME types starting with \texttt{text/} prefix;
\end{itemize}

\section{Asynchronous Processing}
\label{section:async-processing}

Task processing, as pretty much any operation that uses LLMs, is naturally an asynchronous process. LLM requests take several seconds to complete, which means that complex multi-step tasks may take multiple minutes to finish. Design of the system reflects such nature of the problem.

\texttt{POST /tasks} endpoint implements a common convention for long-running operations, returning \texttt{202 Accepted} immediately, rather than waiting for task to complete before returning a response. 

Step completion is implemented using a work item queue, with state saved in the database. When new task is created, initial planning step is added to the queue. Subsequently, when any step is completed, all task steps for which all predecessors were completed are added to the queue, ensuring that task execution continues. This design ensures that the execution is resumable when system is restarted, while also making process thread count independent from the number of running tasks.

\subsubsection{SSE}

HTTP communication is asymmetric by design - it always starts with a request from the client, which the server handles, returning a response. This property makes dealing with long-running asynchronous processes inconvenient - the only way for the client to wait for an operation to complete is to use a pooling loop, checking the state every so often. This problem can be quite easily solved by implementing a non-HTTP notification mechanism (i.e. WebSockets [TODO: ref]). However, a simpler solution exists: Server-Sent Events [TODO: ref]. SSE stream utilizes HTTP for server-sent notifications, by simply keeping the HTTP connection open, writing each event data as a JSON line. While not aligning with HTTP design, this mechanism provides a simple alternative to pooling or using additional connection protocols.

The system exposes an SSE endpoint at \texttt{/sse/events}. Events are produced when task state is updated in any way (step is completed, step list is generated, task finishes, etc.). Additionally, metadata filtering is implemented, making it possible to request only a certain subset of emitted events - for example, only those related to a particular task.

\section{Error Recovery}
\label{section:error-recovery}

LLM-based operations require a relatively extensible error handling. First, regardless of format schemas and specifications, LLM output should be treated as text, which may or may not satisfy requested constraints. Failed format validation during step execution causes the step to be failed, triggering a reevaluation step. 

Second, LLM may simply be unable to complete a step. Steps, just like tasks, aren't constrained in any way, and may specify problems that are extremely hard to solve, or which don't make sense after acquiring certain information from previous steps output or file contents. A plan for a certain mathematical task may contain a step requiring a proof of Riemann Hypothesis. Image recognition task may specify a step requiring a model to determine a scientific name of the highest tree on an attached photo, while the photo does not contain any trees. To solve this problem, step output schema specifies a "failure reason" field, and LLM that executes the step is instructed to use it if the step cannot be completed to specify the problem. Failed step subsequently causes reevaluation to be performed. Reevaluation prompt contains the failure reason, making it possible to plan subsequent steps in a way that solves the problem.

\section{LLM Selection}
\label{section:llm-selection}

Each step requires different LLM capabilities and input modalities to be completed. In addition to these constraints, each viable LLM has different strengths and weaknesses, as well as a different request cost. The goal of LLM selection for a given step is choosing an LLM which satisfies all input modality and capability constraints, while maximizing a certain utility function, combining cost minimization requirement with maximizing prompt-specific expected response quality score.

Ensuring that chosen LLM satisfies input modality and capability constraints is done by simply filtering the list of available models based on their metadata. If no model satisfies these constraints, step is failed with a descriptive reason, and reevaluation is triggered.

\section{Dataset Selection}

The title of the most popular human-evaluated LLM preference dataset without a doubt belongs to \textit{Chatbot Arena Conversations} [TODO: ref]. It is used as primary human-evaluated dataset in referenced works [TODO: verify and ref]. 

\textit{Chatbot Arena Conversations} data consists of a pair of responses from different LLM models to a single user prompt, along with a comparison label, determining which model's answer is considered better by the user. When comparison is unclear, the label can also indicate that both responses were sufficient for a given query, or that neither model managed to produce a good answer. Selection of LLMs is sufficiently wide for a model to learn a broad spectrum of differences between different LLM models, while prompts span a variety of topics. 

However, there is a crucial issue: published in 2022 [TODO: verify], \textit{Chatbot Arena Conversations} dataset only includes comparisons between rather old LLM models. This isn't a disqualifying problem, as my work doesn't focus on particular LLM models. Though, A decision to use this dataset would require two important implications: that human preference during evaluation of a pair of responses remains unchanged, and that the general distribution of responses produced by LLMs is similar.

It's hard to evaluate how close to the truth the first assumption is. While human preferences likely did not change drastically over the last four years, expectations of what LLMs should be able to do did undergo an undeniable change, simply due to their capabilities increasing. Another significant difference may be caused by a wider adoption - people whose responses were used in \textit{Chatbot Arena Conversations} dataset belonged to a significantly narrower group of "early adopters" of LLM technology, compared to a group of LLM users today.

These reasons prompted me to decide to rely on a newer alternative to \textit{Chatbot Arena Conversations} dataset. Collected and released by the same organization (\textit{LMSYS} [TODO: ref]), \textit{LMArena Human Preference} is a successor of aforementioned dataset. It's structure closely resembled \textit{Chatbot Arena Conversations}, also including human comparison results over a pair of LLM responses to the same prompt. Additionally, the dataset includes a range of common prompt labels [TODO: describe], which can be used as additional features in any prediction task. \textit{LMArena Human Preference} is also an improvement in terms of size. It contains around 140 thousand comparison pairs (compared to \textit{Chatbot Arena Conversations}'s 33 thousand), and includes responses from 53 different models (compared to \textit{Chatbot Arena Conversations}'s 20).

It's important to note that due to a very similar structure, these datasets can be used interchangeably. Due to previously mentioned reasons, mainly \textit{LMArena Human Preference} was used for model training and evaluation throughout this work. However, comparisons between datasets are made where relevant, highlighting important differences between their data.

\section{Preprocessing \& Feature Selection}
\label{section:preprocessing-and-features}

Intuitively, when it comes to inferring the quality of LLM's response to a given input, the most important aspect of that prompt should be it's semantic meaning - ideally, prompts that are equivalent when it comes to meaning, differing only in wording, should produce the same answer. Semantic meaning of a given text can be captured via a range of specialized text embedding models, which, given an input, return a vector that should be useful for a range of classification or regression tasks. [TODO: references, examples]. As such, prompt embedding vectors are a primary candidate for model input features.

In practice, assumption that prompt features which are unrelated to it's meaning shouldn't influence LLM output isn't reasonable. LLMs in they current state are far from ideal, so certain biases and simple relationships cannot be ruled out. Thus, instead of relying solely on prompt embedding vectors, input features also contain a set of supporting values that represent certain characteristics of submitted question.

Aside from prompt embedding vectors, the following simple features were used:

\subsection{Complexity Features}

8 features encoding prompt complexity were defined:

\begin{itemize}
    \item \textbf{Character count};
    \item \textbf{Word count};
    \item \textbf{Sentence count};
    \item \textbf{Average word length};
    \item \textbf{Question density} - Number of question marks in the prompt, divided by prompt length;
    \item \textbf{Nesting density} - Number of parentheses, divided by prompt length;
    \item \textbf{Multi-part indicator density} - Number of words indicating multiple aspects/parts in the prompt ("and", "also", etc.), divided by prompt length;
    \item \textbf{Inverse vocabulary richness ratio} - $1 - \frac{\text{[number of unique words]}}{\text{[word count]}}$.
\end{itemize}

Features that don't directly encode prompt length were normalized by that length, in order to reduce dependency between them - for example, it's reasonable to assume that longer prompts will, on average, contain more parentheses and question marks than shorter prompts.

Vocabulary richness ratio was inverted by subtracting it's value from 1. This transformation was performed to make value distribution more similar to other features, which made it possible to unify it's further processing with other features [TODO: ref]. In an overwhelming majority of samples, nearly all words are different, causing vocabulary richness ratio to primarily take a value of 1 (or close to it). "Inverting" values causes them to instead be 0 (or close to it).

\subsection{Non-iterative Models}

The problem of scoring entries based on comparison data can be expressed more rigorously, turning it into an optimization problem with well-defined fitness metric. Some such expressions have easily computable optimal solution.

\subsubsection{L1 Criterion}

The most natural way of expressing quality of score assignment is via accuracy - a percentage of comparisons where winning entry was assigned a higher score than the losing one. This is equivalent to minimizing the number of \textit{upsets} (comparisons such that entry A wins against entry B, but A was assigned a lower score than B). This criterion turns out to be NP-hard [TODO: ref]. However, if we take into account the magnitude of upsets (difference between scores assigned to A and B), the problem becomes solvable in polynomial time. Minimizing the sum of magnitudes can be expressed as a linear programming problem ($v_a$ represents the score assigned to entry A, and $\epsilon$ is added to prevent trivial solution of all scores being 0):

\begin{align*}
    \textrm{minimize} & \quad \sum s_i \\
    \textrm{subject to} & \quad v_a - v_b + s_i \geq \epsilon \\
    \textrm{for} & \quad a > b \quad \textrm{in comparison set}
\end{align*}

This criterion can be equivalently formulated via a min-cost-max-flow problem.

\begin{enumerate}
    \item Each node (except source and sink) represents an entry;
    \item A comparison in which A wins against B is expressed as an edge from node A to node B, of capacity 1 and cost 1;
    \item There's an edge from source to each node, with cost 0 and capacity equal to it's number of wins;
    \item There's an edge from each node to sink, with cost 0 and capacity equal to it's number of losses.
\end{enumerate}

Then, node potentials can be interpreted entry scores.

Any cycle in the network has positive cost (since all edges that don't include source and sink have positive cost), so optimal flow won't use them. Using that information, we can compute the ratio of used comparison edges (equal to total cost) to all comparisons. This quantity can be interpreted as decisiveness of comparison set - since comparison cycles can't be scored correctly, the more common they are, the less correct is it to represent LLM quality via a single, prompt-independent score.

\section{LLM Embedding}
\label{section:llm-embedding}

LLM ecosystem is constantly evolving, with new models being introduced often. While dataset selection was made with that in mind, and chosen dataset contains comparisons of LLMs that are quite new at the time of writing, constraining scoring models to only work with LLMs featured in a comparison dataset doesn't seem like a practical decision. Such datasets rely on human evaluation, and aren't easy to prepare.

An extensible alternative to scoring only LLMs whose comparison data is available, is to compute an embedding vector representing each model, and use that instead of LLM ID as scoring input data. Such embeddings can be computed without relying on comparison data, making the model easily extensible. In particular, embeddings can be computed from a set of prompts and LLM responses to them. This information is much easier to gather than human evaluations, and can be collected automatically. 

An argument can be made that prompts and responses represent all information required for scoring. This way of specifying the problem transforms it into a set embedding problem: given a set of samples, define an embedding function that satisfies some requirements, while being invariant to sample ordering. Many solutions to this problem have been developed and discussed. One such solution is described by Zaheer et. al. in \textit{DeepSets} paper \cite{zaheer2017deep}. It shows that such set embedding function can be defined as a function of a mean of sample embeddings (equation \ref{eq:set-embedding-mean}). $P_{out}$ represents final output transformation, while $E_s$ is a sample embedding function. 

\begin{equation}
\label{eq:set-embedding-mean}
    E(S) = P_{out}(\frac{1}{\vert S \vert} \sum_{x \in S} E_s(x))
\end{equation}

This specification of the problem has an important, additional benefit. Comparisons dataset which was selected contains prompts and responses from 50 distinct LLMs. Original formulation of the problem, which focuses on assigning a single embedding vector to each LLM, only ever produces 50 different embedding values, which severely limits the dimensionality of embedding vectors before the problem becomes underspecified. By viewing LLM characteristics as only defined by responses they produce to given prompts, we can instead focus on computing embedding vectors for each prompt and response pair, raising the number of computed embedding values to ~140k.

\subsection{Attention Embedding}

Set embedding problem is quite common in machine learning, and as such, a number of solutions were proposed over the years. The general idea presented in the preamble of section \ref{section:llm-embedding}, defining LLM embedding as a mean of embeddings of their outputs, is loosely inspired by Zaheer et. al. \textit{DeepSets} paper \cite{zaheer2017deep}. While well received and widely cited, alternatives and extending works exist. One such work is Lee et. al. \textit{Set Transformer} paper \cite{lee2019set}. It challenges an implicit assumption that importance of all set elements is equal. The paper proposes a relatively simple alternative to taking a mean of sample representations - attention mechanism. The rationale behind that change is that certain samples provide a much clearer information about the set than others - capturing behaviors or characteristics that highlight differences between a given set and the rest.

Attention embedding model implements that idea. The model consists of two modules: pair encoder (illustrated by figure \ref{fig:attention-embedding-pair-encoder}) and set aggregator (shown on figure \ref{fig:attention-embedding-set-embedding}). Pair encoder takes prompt and response embeddings together with scalar features as inputs, projects each into a vector of configurable dimensionality, enriches these projections with interaction features (difference between prompt and response embeddings, and their element-wise product), then runs these features through a multi-layer perceptron to produce an embedding vector.

The model can be described by equation \ref{eq:attention-embedding}. $E_s$ is the sample embedding function, $P_{out}$ - final projection module, $K$ - key projection (linear), $V$ - value projection (linear), and $Q$ - query vectors (trainable parameters).

\begin{align}
\label{eq:attention-embedding}
    E(S) &= P_{out}(Attention(E_s(S))) \\
    Attention(X) &= softmax(\frac{Q K^T(X)}{\sqrt{dim(Q)}}) V(X)
\end{align}

Set aggregator module uses sample embeddings produced by pair encoder, outputting a single LLM embedding vector. In previous embedding models, this step was done by a simple mean computation, instead. Set aggregator uses attention mechanism. Pair embeddings are used to produce key ($K$) and value ($V$) vectors through learnable linear projections. Then, attention masks are computed by a configurable number of attention heads. This mask is then used to produce a single LLM embedding from value vectors produced from pair embeddings.

\subsection{DNN Model}
\label{section:dnn-model}

The first deep learning model that was implemented uses a simple multi-layer neural network. It takes LLM embeddings, prompt embeddings, and prompt scalar features as input, and outputs a score representing how well given LLM would handle provided prompt.

Three distinct input vectors don't have to share any characteristics, like dimensionality, scale, or importance. For each, a separate projection layer was introduced. These layers transform input vectors into representations of set dimensionality, which subsequently are concatenated and used as input to the following layers of the network.

Number and sizes of hidden layers are configurable. \textit{LeakyReLU} was chosen as the activation function, with negative slope set to $0.1$. Dropout layers were introduced between each pair of hidden layers to prevent overfitting. Additionally, \textit{LayerNorm} layers were introduced for regularization, and gradient clipping was introduced to prevent gradient explosions during training.

The score assignment goal that the network tries to achieve does not inherently require any specific scale of outputs. As scores are only ever used for comparison, transforming them through any strictly increasing function does not change the quality of the result - for each compared pair of models, bigger score will still be bigger after the transformation. This ambiguity is undesired and impractical from human perspective. Introducing a secondary training objective could be a solution to this problem, however it is unnecessary in this case. Instead of introducing an additional loss term, whose influence may negatively impact the primary objective, score post-processing was introduced. After training, mean and standard deviation of scores assigned to training data is computed, such that subsequent inference results can be normalized to mean 0 and standard deviation equal to 1. Additionally, scaled scores are transformed using a scaled hyperbolic tangent function onto a $(-1, 1)$ range.

\subsubsection{\textit{AdamW}}

\textit{AdamW} optimizer was used during training process. Described by Loshchilov et. al. in \textit{Decoupled Weight Decay Regularization} paper \cite{loshchilov2019adamw}, it improves the previous \textit{Adam} algorithm (Kingma et. al., \textit{Adam: A Method for Stochastic Optimization} \cite{kingma2015adam}) by decoupling weight decay from gradient update step. In \textit{Adam}, L2 regularization is essentially implemented by altering the effective loss criterion which is being minimized. \textit{AdamW} implements weight decay during weight update step instead, as described by equation \ref{equation:adamw-weight-update}. 

\begin{equation}
\label{equation:adamw-weight-update}
    \theta_{t+1} = \theta_t - \alpha \frac{\hat{m}_t}{\sqrt{\hat{v}_t} + \epsilon} - \alpha \lambda \theta_t
\end{equation}

\subsubsection{Residual Connections}

As depth of a dense neural network increases, a problem of vanishing gradient arises. [TODO: reference]. Gradient of early layer weights is a product of activation function gradients and weight values of later layers. Due to regularization and similar mechanisms, these values tend to be smaller than 1 on average, causing gradual decrease in gradient magnitude. To mitigate this issue, residual connections were introduced. Initially described by He et. al. in \textit{Deep Residual Learning for Image Recognition} paper \cite{he2016deep}, residual connections introduce a "skip-path" for each layer. Mathematically, this means that layer input is added to layer output. The modification ensures that early layers have a direct path to network output. As vector addition can only be performed if vectors have the same dimensionality, the implementation of skip connections that relies on adding layer input to output can only be performed for layers that have the same input and output dimension. The mechanism can be easily extended to layers where these dimensions are not equal, though, by introducing a linear projection layer with the same input and output dimensions as the "primary" layer, but without activation functions, normalization layers, and so on. This extension is also described in \textit{Deep Residual Learning for Image Recognition} paper \cite{he2016deep}, in the context of skip connections between CNN layers with different channel counts. The idea can be justified by the fact that application of multiple linear transformations is equivalent to a single linear transformation. In other words, an arbitrary early layer of the network can influence the output through a single linear transformation, which means it receives a clear gradient signal. 

\subsubsection{Loss Functions}

Two interchangeable loss functions were implemented: margin ranking loss (defined by equation \ref{equation:margin-ranking-loss}) and Bradley-Terry sigmoid cross-entropy loss (defined by equation \ref{equation:cross-entropy-loss}). $s_a$ and $s_b$ represent assigned scores, $y \in \{-1, 1\}$ - comparison result, $\sigma$ is the sigmoid function, and $BCE$ is a binary cross-entropy loss function.

\begin{equation}
\label{equation:margin-ranking-loss}
    \mathcal{L}_\text{margin ranking}(s_a, s_b, y) = \frac{1}{n}\sum_{i=1}^{n} \max\bigl(0;-y_i(s_{a,i}-s_{b,i}) + m\bigr) 
\end{equation}

\begin{equation}
\label{equation:cross-entropy-loss}
    \mathcal{L}_\text{cross-entropy}(s_a, s_b, y) = \frac{1}{n}\sum_{i=1}^{n} \text{BCE}\left(\sigma(s_{a,i}-s_{b,i}), \frac{y_i+1}{2}\right)
\end{equation}

Initially, margin ranking loss seemed more favorable - once scores assigned to a given training pair satisfy the comparison with a margin, the pair has no contribution to loss gradients, and therefore model is not encouraged to widen the score gap. Intuitively, this is a desired behavior. Assigning a widely different scores to compared samples is by no means better than keeping their difference small. However, margin ranking loss turned out to under-perform significantly in practice. This observation motivated the implementation of an alternative.

\subsubsection{Metrics}
\label{section:dnn-model-metrics}

Training process of every machine learning model can be monitored by following two primary metrics: loss function values, which the model is trained to minimize, and accuracy values, which, assuming good design of loss criterion, should increase as loss values decrease. Recording these values after every training epoch for both training and validation sets provides a way to verify if the model is learning correctly, overfits, stagnates, etc.

Although undeniably useful, accuracy and loss values alone are not enough to interpret the behavior of the model and to diagnose it's flaws. A set of additional diagnostic metrics was introduced to help with that. They were chosen specifically to provide information about internal network behavior. The following metrics were defined:

\begin{itemize}
    \item \textbf{Projection variance and norm} \\
    Variance and L2 norm of projected values of each input component (outputs of their projection layers) are tracked. Low variance of component projections signifies it's low impact on model output - the network essentially learned to ignore any information that could be contained in that input component. Additionally, relative magnitude of projection norms can provide information about relative impact of different input components on network output. 
    \item \textbf{Score variance decomposition} \\
    Variance of assigned scores is decomposed into LLM-specific and prompt-specific components. Decomposition can be described by equation \ref{equation:variance-decomposition} - $s$ represents assigned scores, $\overline{s}_m$ - mean scores of LLMs, and $\Delta s$ - deviations of assigned scores from the average LLM score. If nearly all variance in returned scores can be attributed to LLM differences (or, more specifically, to LLM embeddings), we can conclude that prompt information is essentially ignored. \\
    \begin{equation}
    \label{equation:variance-decomposition}
        Var(s) = Var(\overline{s}_m) + Var(\Delta s)
    \end{equation}
    \item \textbf{Input gradient attribution} \\
    During training, loss criterion gradients are computed with respect to inputs. Dot product of those gradients and input values can be used to measure importance of each input value - described by equation \ref{equation:gradient-attribution}. This metric is especially useful, because it can be computed not only for input components, but also for individual coordinates in a given input component vector. While per-coordinate importances don't make much sense in case of LLM and prompt embeddings, they do provide a lot of information in case of scalar features - features that are not useful for scoring can be easily identified by their low importance. \\
    \begin{equation}
    \label{equation:gradient-attribution}
        Imp(x) = \nabla_x L \cdot x
    \end{equation}
\end{itemize}

\subsubsection{Post-training Sensitivity Analysis}

In addition to metrics described in section \ref{section:dnn-model-metrics}, which are computed and tracked per epoch, additional post-training sensitivity metrics were implemented. While the primary goal of previously defined metrics is training process monitoring and diagnosis of certain undesired patterns, post-training metrics described in this section are implemented to provide an interpretation of the scoring behavior of a trained model. While it would be possible to compute these metrics after every epoch as well, to provide an insight into how scoring behavior is changing during training, doing so proved impractical due to relatively high computational cost. Measuring each metric requires at least one additional pass through the training set.

The following sensitivity metrics were implemented:

\begin{itemize}
    \item \textbf{Prompt sensitivity} \\
    Prompts are randomly permuted across all samples, and model accuracy is measured. Drop in accuracy can be interpreted as model's sensitivity to prompt information - if the model learned to rely on LLM embeddings, the accuracy won't change significantly, and if the model does consider prompt inputs, the drop will be significant.
    \item \textbf{Prompt embedding sensitivity} \\
    Model accuracy is measured after setting all prompt embeddings to their mean across the sample set. Drop in accuracy can be interpreted as model's sensitivity to prompt embeddings, specifically.
    \item \textbf{Scalar features sensitivity} \\
    Model accuracy is measured after setting all scalar feature inputs to their mean across the sample set. Drop in accuracy can be interpreted as model's sensitivity to scalar features.
    \item \textbf{Per-feature sensitivities} \\
    Model accuracy is measured after randomly permuting values of a single scalar feature across the sample set. Drop in accuracy can be interpreted as model's sensitivity to that feature.
\end{itemize}

Each metric is computed separately on training and validation sets.

\subsubsection{Interaction Features}

The model is designed to learn most information required for score prediction through fine-tuning. This decision dictates the small size of scoring head module - it's only purpose is to combine LLM embedding vectors (and scalar prompt features, if they turn out to be useful) with prompt embeddings. However, this introduces certain risks. The "path length", or, more formally, number of operations, between prompt text and output is significantly bigger than between other inputs and output. This may cause the model to initially take the path of least resistance, focusing only on LLM embeddings (essentially learning optimal prompt-invariant score for each vector), and discarding prompt embeddings entirely, essentially blocking gradient flow to the transformer module.

To address this issue, simple concatenation of projections of scoring head input vectors was replaced by another combining operation. The goal is to discourage the model from learning scores that depend only on LLM embeddings, and instead encourage it to produce scores based on certain interactions between prompt representation and LLM embedding. To encode that interaction, scalar product was used. Scoring head module takes as input the scalar product of LLM embedding projection, and concatenated prompt embedding and scalar prompt features projections. As scalar product requires arguments to be of the same dimensionality, projection layer sizes were adjusted. Size of scalar features projection is configurable, with prompt embedding projection being twice as big, and LLM embedding projection - three times as big.

Of course, this approach does not guarantee that the model cannot ignore prompts. [TODO: handle zeroing prompt text input while keeping prompt features input]. To achieve that, while retaining LLM embedding information, transformer module would need to learn to output a constant, non-zero vector for every prompt. Importantly, though, this state doesn't block the gradient flow to transformer module - even if the model initially does learn to ignore prompt information, it should, theoretically, be able to continue learning prompt-specific relationships after reaching optimal prompt-invariant scoring accuracy.

\subsubsection{\textit{Muon} optimizer}

Instead of \textit{AdamW} optimizer, which was used for dense network model training, transformer model uses \textit{Muon} optimizer. Introduced by Jordan et. al. in an article titled \textit{Muon: An Optimizer for Hidden Layers in Neural Networks} \cite{jordan2024muon}, and later evaluated in context of LLM training by Liu et. al. in \textit{Muon is Scalable for LLM Training} paper \cite{liu2025muonscalable}, \textit{Muon} changes how momentum influences weight updates. While \textit{AdamW} only uses element-wise operations to update weights, \textit{Muon} exploits the matrix structure of most network parameters.  

Algorithm \ref{algorithm:muon} describes weight update process. After gradient is computed and momentum is updated, orthogonalization is performed on the result by computing a polar factor $X = U V^T$, from singular value decomposition $M = (U V^T)(V \Sigma V^T)$. The result $X$ is a semi-orthogonal matrix, which captures the direction of required weight updates while normalizing the magnitude. This step effectively replaces the per-element second moment estimate ($\hat{v}_t$) normalization in \textit{AdamW} algorithm. Additionally, resulting matrix $X$ has all singular values equal to 1, so weight update $O_t$ has a bounded spectral norm regardless of how large the gradient was. This means that orthogonalization acts as implicit regularization.

Before orthogonalization is performed, Nesterov lookahead momentum correction is applied (Y. Nesterov, \textit{Introductory Lectures on Convex Optimization: A Basic Course} \cite{nesterov2004introductory}). As magnitude information is discarded, direction accuracy becomes more important, making the correction more important compared to previous optimizing algorithms, like \textit{Adam}.

Orthogonalization is performed using Newton-Schulz iterative method. It is an approximate, inversion-free alternative to computationally expensive exact singular value decomposition methods. 

\begin{algorithm}[h]
\caption{Muon optimizer (for 2D hidden-layer parameters)}
\label{algorithm:muon}
\begin{algorithmic}[1]
\REQUIRE learning rate $\eta$, momentum $\mu$, weight decay $\lambda$, NS steps $K=5$, NS coefficients $(a,b,c)$
\STATE Initialize momentum buffer $M_0 \gets 0$
\FOR{$t = 1, 2, \ldots$}
    \STATE $G_t \gets \nabla_\theta \mathcal{L}(\theta_{t-1})$ \hfill \COMMENT{compute gradient}
    \STATE $M_t \gets \mu \, M_{t-1} + G_t$ \hfill \COMMENT{momentum accumulation}
    \STATE $\hat{M}_t \gets M_t + \mu \, (M_t - M_{t-1})$ \hfill \COMMENT{Nesterov lookahead (optional)}
    \STATE $X \gets \hat{M}_t$ \hfill \COMMENT{begin Newton--Schulz orthogonalization}
    \STATE $X \gets X \,/\, \|X\|_F$ \hfill \COMMENT{normalize for NS convergence}
    \FOR{$k = 1, \ldots, K$}
        \STATE $X \gets a X + b (X X^\top) X + c (X X^\top)^2 X$
    \ENDFOR
    \STATE $O_t \gets X$ \hfill \COMMENT{$O_t \approx U V^\top$ from polar decomposition of $\hat{M}_t$}
    \STATE $\theta_t \gets \theta_{t-1} - \eta \, O_t - \eta \lambda \, \theta_{t-1}$ \hfill \COMMENT{update with decoupled weight decay}
\ENDFOR
\end{algorithmic}
\end{algorithm}

\subsection{Response Predictive Model}

Models described in previous sections have one common characteristic: they do not use LLM responses during training, despite them being available in the dataset. The reason of that is quite obvious - LLM responses are not accessible at inference time. The whole point of assigning scores to LLMs is to be able to decide which will produce the best response, without actually requesting that response. Intuitively, if LLM responses were available, the task should be significantly easier. Assigning scores based only on the prompt and information about chosen LLM internally requires deducing certain response characteristics, based on which scoring can subsequently be performed.

The main idea of response predictive model is to utilize response data as much as possible, without requiring it for inference. This is achieved by dividing the scoring task into two parts, described in previous paragraph. First module, the predictor, is responsible for predicting response representation, given information about the prompt and chosen LLM. Second module, the scorer, is responsible for assigning a score, based only on that representation (without additional information about the LLM). End-to-end process retains the same input/output interface as other models, taking prompt and LLM data as input, and returning a score. However, explicit separation of evaluation steps opens new possibilities for training.

Prompt and LLM representations used by response predictive model are the same as in other models. Prompts are represented by text embedding vectors and scalar features, while LLMs are encoded as embedding vectors. As this model also uses LLM responses, they need to be represented somehow. The simplest possibility would be to use a similar representation to what is described in section \ref{section:llm-embedding-features}. This representation might be useful for scoring, however there is no guarantee that it would be possible to predict it from just prompt and LLM data. In general, predicting LLM responses is an extremely hard task - predicting all characteristics of the response would be essentially equivalent of generating the response, requiring parameter counts and training data volumes orders of magnitude larger than what's achievable in this project. So, response representation should be carefully tuned in such a way that it is still useful for the scoring module, while also being predictable by the predictor module.

Finding such a representation can be incorporated into the training process. A third module is introduced: response encoder. Given response representation (in format described in section \ref{section:llm-embedding-features}), it learns to produce response embedding vectors.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\textwidth]{thesis/diagrams/response-predictive-model.drawio.png}
    \caption{Response predictive model architecture}
    \label{fig:response-predictive-arch}
\end{figure}

Importantly, predictor and response encoder operate on different inputs. Response encoder has access only to real LLM response representations, while predictor has no information about them. As responses are not available during inference, response encoder is not used in that case. It's purpose is to help in discovering response representations that can be useful for scoring. Having access to actual responses, finding such embeddings should be much easier.

Modular structure of response predictive model makes it more interpretable than most other models, which usually function as black-box predictors. Each of the three modules has it's separate role, making it easy to measure the quality with which it performs it's task. In particular, we can decouple prediction errors from representation and scorer errors, by comparing scoring accuracy when response embeddings are predicted, with scoring accuracy using real response embeddings. Similarly, predictor quality can be tracked by measuring common regression metrics, like mean squared error or cosine similarity, and by comparing distributions of real and predicted embeddings - ideally, if predictor and encoder modules find some representation equilibrium, distribution should be similar.

\subsubsection{Loss Function}

Due to relative complexity of training process of response predictive model, it requires a more complex loss function design than other models. Using a simpler design (i.e. just ranking loss) ultimately leads to a "collapse" to a simpler, locally optimal state. For example, if no effort is made to ensure predicted and encoded representations are similar, scoring module can just learn to treat both types of representations differently. 

Four loss components are combined, according to equation \ref{equation:response-predictive-loss}. Weights $k_\text{pred}$, $k_\text{repr}$ and $k_\text{KL}$ are configurable.

\begin{align}
\label{equation:response-predictive-loss}
    \mathcal{L}_\text{response predictive} = \mathcal{L}_\text{rank} & + k_\text{pred} \mathcal{L}_\text{pred} + w(n_\text{epoch}) k_\text{repr} \mathcal{L}_\text{repr} + \\
    + k_\text{anchor} \mathcal{L}_\text{anchor} & + k_\text{consistency} \mathcal{L}_\text{consistency} + k_\text{dist} \mathcal{L}_\text{dist}
\end{align}

First criterion is ranking loss. Two interchangeable ranking loss options were defined, the same as in previously described models: margin ranking loss (equation\ref{equation:margin-ranking-loss}) and Bradley-Terry sigmoid cross-entropy (equation \ref{equation:cross-entropy-loss}). Loss criterion is computed not only using predictor module flow (where response representations are predicted), but also through encoder flow, using real responses as input.

Second criterion encodes prediction accuracy. It encourages predictor module to produce outputs similar to those produced by response encoder. Encoder outputs are treated as constants in this case - gradients are computed only with respect to predictor weights. Third criterion is the same as second, but treating predictor outputs as constants instead. It instructs response encoder module to produce representations that predictor module is able to predict. Three interchangeable variants were defined for this loss component: MSE (equation \ref{equation:mse-loss}), cosine similarity (equation \ref{equation:cosine-similarity-loss}) and Huber L1 criterion (equation \ref{equation:huber-l1-loss}). $\hat{r}$ represents predicted response representations, and $r$ - encoded response representations. The sum in Huber criterion iterates over response vector components. 

\begin{equation}
\label{equation:mse-loss}
    \mathcal{L}_\text{MSE} = \frac{1}{n} \sum_{i=1}^{n} \bigl|\hat{r}_{i} - r_{i}\bigr|^2
\end{equation}

\begin{equation}
\label{equation:cosine-similarity-loss}
    \mathcal{L}_\text{cosine} = \frac{1}{n} \sum_{i=1}^{n} \bigl(1 - \frac{\hat{r}_i \cdot r_i}{|\hat{r}_i| |r_i|}\bigr)
\end{equation}

\begin{equation}
\label{equation:huber-l1-loss}
    \mathcal{L}_\text{huber} = \frac{1}{d_r}\sum_{j=1}^{d_r} \\
    \begin{cases} 
        \tfrac{1}{2} (\hat{r}_{i,j} - r_{i,j})^2 & |\hat{r}_{i,j} - r_{i,j}| \le 1 \\
        |\hat{r}_{i,j} - r_{i,j}| - \tfrac{1}{2} & \text{otherwise} 
    \end{cases} 
\end{equation}

Predictor loss component is additionally weighted by function $w(n) = \max(1, \frac{n}{n_w})$, where $n$ is the current epoch number. At the beginning of training, response representations produced by predictor module are highly random. There's no point in training response encoder to produce outputs that are similar to such random noise. $w(n)$ gradually ramps up the importance of that loss criterion as training progresses and predictor starts learning. 

Fourth criterion is given by equation \ref{equation:kl-anchor-loss}. It is a KL loss term, encouraging a normalized distribution of response embeddings. In particular, this prevents initial representation collapse to a single value - it is a trivially optimal solution to the second and third loss criterion. Importances of loss criteria are controlled by configurable weights. [TODO: cite something + explain]

\begin{equation}
\label{equation:kl-anchor-loss}
    \mathcal{L}_\text{anchor} = \frac{1}{d_r}\sum_{j=1}^{d_r} \frac{1}{2}\left(\sigma_j^2 + \mu_j^2 - 1 - \ln\sigma_j^2\right) 
\end{equation}

Fifth criterion is given by equation \ref{equation:score-consistency-loss}. It encourages both encoder and predictor to produce representations that would be scored similarly by the scorer model.

\begin{equation}
\label{equation:score-consistency-loss}
    \mathcal{L}_\text{consistency} = \frac{1}{n} \sum_{i=0}^{n} (Score(\hat{r}_i) - Score(r_i))^2
\end{equation}

Last criterion is given by equation \ref{equation:distribution-loss}. It is a sum of Jeffrey's divergences between encoded and predicted values of each response representation component. [TODO: cite something + explain]

\begin{align}
\label{equation:distribution-loss}
    \text{KL}\bigl(\mathcal{N}(\mu_1,\sigma_1^2)|\mathcal{N}(\mu_2,\sigma_2^2)\bigr) & = \frac{1}{2}\left(\frac{\sigma_1^2}{\sigma_2^2} + \frac{(\mu_1-\mu_2)^2}{\sigma_2^2} - 1 + \ln\frac{\sigma_2^2}{\sigma_1^2}\right) \\
    \mathcal{L}_\text{dist} & = \frac{1}{d_r} \sum_{i=1}^{d_r} \frac{\text{KL}(\hat p_j | p_j)+\text{KL}(p_j| \hat p_j)}{2}
\end{align}

\subsection{Multistage Models}

Models described in this section are focusing on assigning a score based on prompt and LLM information. In some cases, it's possible to at least partially interpret their decisions ([TODO: ref]), but in general this formulation of the task does not imply any interpretability. To help with that, model chaining mechanism was introduced. Taking advantage of the fact that scores learned during training are not constrained in any way, given a trained scoring model as "base", another can be fitted on it's errors, attempting to predict the difference between it's predecessor and the true value. Final scores are defined as the sum of both predictions.

We can describe model chaining via equation \ref{equation:model-chaining}. The main interpretability benefit comes from a possibility of constraining input values passed to each model - $I_k$ for a model given by $Score_k$. By gradually extending the set of input components, impact of introduction of each component is clearly visible as an increase in joint model performance quality. 

\begin{equation}
\label{equation:model-chaining}
    Score(I) = \sum_k Score_k(I_k), \quad \forall_k I_k \subset I
\end{equation}

In LLM scoring task, multi-stage models can be especially useful when a prompt-invariant model is used as base. As described in section \ref{section:prompt-invariant-scoring}, optimal score assignment based only on chosen LLM is rather simple. By using such model and training a more complex model to predict prompt-specific differences, we clearly separate accuracy and score contributions that use prompt information from base LLM-only "bias".

An additional advantage of multi-stage approach is faster training. Complex models may take multiple epochs before reaching the optimal prompt-invariant accuracy. If prompt-invariant model is used as base, however, optimal prompt-invariant accuracy is the starting point.
